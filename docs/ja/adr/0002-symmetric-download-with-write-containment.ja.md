# ADR-0002: 対称 download と書き込み側封じ込め

> Status: Accepted — 2026-07-20

## 背景

純正 Slack MCP はファイル内容を*モデルのコンテキストへ*読む
（`slack_read_file`）が、Slack のファイルを**ローカルディスクに置く**手段は
ない。バイナリ・アーカイブ・コンテキストに収まらない大きさのファイルを
エージェントが実際に扱うのはディスク上である。upload だけでは非対称で、
拡張はローカル成果物を Slack へ送れるのに、共有され返ってきたファイルを
取得できない。

脅威モデル: upload は **egress** プリミティブ（exfil リスク — 読み取り側
ポリシーで封じ込め済み）。download はその鏡像の **ingress/書き込み**
プリミティブであり、非信頼な Slack コンテンツに誘導されたエージェントが、
許可領域の外への書き込み・既存ファイルの上書き・Slack 由来ファイル名
（`../../evil`, `.zshrc` 等）による敵対的パスの持ち込みを試み得る。
Slack ファイルの filename フィールドは攻撃者が制御可能なデータである。

## 決定

`ext_file_download`（file_id → ローカルファイル）を追加し、同一封じ込め
ポリシーの書き込み側変種で統制する。全チェックは canonical 化後のパスに
対して行う:

1. **書き込み先の包含**: 書き込み先の*親ディレクトリ*が実在し、canonical 化
   （Abs+Clean+EvalSymlinks）の結果 allowed root 配下にあること。root は
   upload と同一の deny-by-default 集合で、書き込み先指定が範囲を広げる
   ことはない。
2. **ファイル名サニタイズ**: 実効 basename（エージェント指定 `filename`、
   なければ Slack ファイル名）は base 成分に還元し、区切り文字や親参照は
   残らない。隠し成分ルールも適用する（dotfile は書けない）。
3. **上書き拒否**: 既存ターゲットは構造化エラー `already_exists` で失敗し、
   エージェントが別名を選ぶ。v1 に上書きフラグは設けない — 削除は本ツールの
   責務ではない。
4. **サイズ上限**: 設定の max_file_size を適用。files.info の申告サイズを
   転送前に検査し、書き込み中もストリームをハードリミットする（申告サイズの
   偽装では上限を回避できない）。
5. **アトミック書き込み + ingress 監査**: tmp+rename で書き込み、ok/denied/
   error の全試行を upload と同一の監査ログに記録する（方向はツール名で
   自明）。

実装: files.info（メタデータ + url_private_download）→ 認証付き GET →
封じ込め済み書き込み。`files:read` scope が必要（推奨 scope セットに含有済）。

## 帰結

- upload と download が機能・統制の両面で対称になり、operator の
  `allowed_roots` が双方向のファイルフローを束縛する。
- Slack 側のファイル名がファイルの*置き場所*に影響することはなく、
  （サニタイズ後の）*名前*にのみ影響する。
- テキスト内容が欲しいだけなら公式 `slack_read_file` を使うべきで、
  `ext_file_download` は実ファイルをディスクに得るためのツールである。
